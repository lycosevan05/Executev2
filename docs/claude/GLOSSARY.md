# GLOSSARY.md — Execute (Executev3)

Project-specific terms, defined so nothing is ambiguous in a Project chat. Each
entry is a tight definition plus, for code terms, where it lives. Ordered
alphabetically. Read [`REPO_MAP.md`](./REPO_MAP.md) for orientation,
[`DATA_MODEL.md`](./DATA_MODEL.md) for the schema, [`KEY_FLOWS.md`](./KEY_FLOWS.md)
for runtime behavior, and [`CONVENTIONS.md`](./CONVENTIONS.md) for house style;
this doc only defines vocabulary.

> Scope note: terms are grounded in the repo or the four sibling docs. Where a
> term's meaning can't be pinned down from code, that is said explicitly.

---

### `@/` alias
The Vite import alias for `src/` (`vite.config.js`: `@ → ./src`). Cross-module
imports use it (`import { backend } from '@/api/backendClient'`) instead of deep
relative paths. The two shim files (lucide, recharts) are the deliberate
exception — they use relative paths into `node_modules`.

### ADDITIVE_FIELDS
The set of vital categories that *accumulate* across a day rather than overwrite:
`steps`, `sleep`, `water`. Defined in `src/lib/vitalsLog.js`; used by
`getDailyLogUpdatesForCategory` so logging "+2000 steps" adds to the day's total.

### AIPlan / master plan
The single canonical AI-generated plan record per user — a 7-day **overview**
(plan summary, nutrition targets, training split, recovery strategy,
`weekly_overview` with exactly 7 days), created by one LLM call. Entity `AIPlan`
→ table `ai_plans`. Created in `src/lib/generateInitialPlanBundle.js`; creating a
new one archives the prior `status: 'active'` plan to `'archived'`. The overview
deliberately contains **no** detailed daily workouts/meals — those are generated
lazily per day.

### appCache
The two-tier stale-while-revalidate cache for cold-launch correctness, in
`src/lib/appCache.js`. Tier 1 is an in-memory `Map` (`STORE`, the only thing
synchronous `get`/`isFresh` read); Tier 2 is durable (`durableStore`). Key API:
`whenHydrated()`, `activateUser(uid)`, `deactivate()`, `set`/`setForUser`,
`invalidate`. Distinct from the lightweight module-level TTL caches and from
TanStack Query. See also: bootHydrate, loading floor, durableStore.

### appUserID (RevenueCat)
The identity RevenueCat keys a customer on. Execute sets it to the user's
**Supabase email** (`Purchases.logIn({ appUserID: email })` in
`src/lib/revenuecat.js`). That is what makes the webhook's `event.app_user_id`
equal the `user_id` column server-side.

### bootHydrate
The function (`src/lib/appCache.js`) enqueued as the **first** op at module init
that replays the last-active user's durable cache entries into the in-memory
`STORE` before any React effect runs. Always settles `hydrated = true` (even on
empty/failed reads or a 3s timeout) so the loading floor can never wedge.

### BYO ("bring your own plan", `planType: 'custom'`)
The 4th questionnaire option: the user pastes or uploads (PDF) their own training
and/or nutrition plan; Execute structures it and builds the missing side around
it. **No DB schema change** — everything persists inside existing JSON columns
(`plan_payload.byo_*` and `weekly_overview.days[i].byo_session/byo_meal_focus`).
Inherits the premium gate. Key files: `src/lib/plans/byoDraft.js`,
`structurePastedPlan.js`, `extractPdfText.js`, `byoCadence.js`,
`PlanQuestionnaire.jsx`, plus BYO branches inside `generateInitialPlanBundle.js`.
See KEY_FLOWS Flow 6.

### byoCadence / resolveByoSession / resolveByoMealFocus
`src/lib/plans/byoCadence.js` — normalizes a BYO plan's activity level and
resolves which session/meal-focus a given calendar day maps to (honoring
`rest_weekdays` and an `advance` of `'daily'` | `'training_days_only'`). Lets a
short pasted plan repeat correctly past the 7-day overview.

### byoDraft / crash-draft
`src/lib/plans/byoDraft.js` persists the in-progress BYO paste to `durableStore`
under `byo:draft:u:<uid>` (local-only) so an app kill mid-questionnaire doesn't
lose the typed/pasted text. Cleared (`clearByoDraft()`) only on generation
success.

### `bust<Thing>Cache()` / `reset<Thing>Caches()`
Naming idiom for the manual cache invalidators paired with module-level TTL
caches (e.g. `bustSubscriptionCache()` in `src/lib/subscription.js`,
`resetPersonalizationCaches()` in `src/lib/personalizationSync.js`).

### Capacitor (SPM-based)
The native shell wrapping the React/Vite web app into an iOS app. Capacitor 8,
**Swift Package Manager-based with NO Podfile/CocoaPods** — native plugins live
in `ios/App/CapApp-SPM/Package.swift` (App, Browser, Preferences, RevenueCat
Purchases). Config is `capacitor.config.ts` (TypeScript; there is no `.js`
variant); the `.json` in `ios/App/App/` is generated by `cap sync`.

### `cn(...)`
The canonical className combiner, `twMerge(clsx(inputs))`, in `src/lib/utils.js`.
Used by all `ui/` primitives to compose Tailwind classes.

### durableStore
`src/lib/durableStore.js` — async `getItem`/`setItem`/`removeItem`/`keys` routed
by `isNative()`: Capacitor **Preferences** on iOS, `localStorage` on web. Backs
appCache Tier 2 and survives a true iOS app kill (the old `sessionStorage` tier
did not — that was the cold-launch flash cause). **Gotcha:** `Preferences` is
imported *statically*; a dynamic `import('@capacitor/preferences')` hangs forever
on iOS `capacitor://localhost`.

### EntityClient
The class in `src/api/backendClient.js` that wraps each entity table with
`list`/`filter`/`create`/`update`/`delete`/`subscribe`. Exposed as
`backend.entities.<Entity>`. All application data access goes through it — app
code never touches the raw Supabase client for table CRUD (that's reserved for
auth + edge functions).

### entity tables (the 19)
The 19 generic tables that share one templated 8-column definition — all
entity-specific fields live untyped inside a single `data jsonb` column, *not* as
real columns. Listed in the `TABLES` map (`backendClient.js`): `AIPlan`,
`CustomChecklistItem`, `DailyLog`, `FoodLog`, `Goal`, `GoalProgressEntry`,
`InjuryProfile`, `MealPlan`, `NutritionProfile`, `ReadinessCheckIn`,
`SavedRecipe`, `User`, `UserAIContext`, `UserPageLayout`, `UserProfile`,
`UserSubscription`, `WorkoutLog`, `WorkoutPlan`, `WorkoutProfile`. Contrast with
`house_listings` (the only real-column table). See DATA_MODEL §2–3.

### entitlement ("premium")
The RevenueCat entitlement that gates the app. Offering is `"default"` (must be
"Current" in the RC dashboard); package IDs `$rc_monthly` →
`execute_premium_month`, `$rc_annual` → `execute_premium_annual`. Whether a user
*has* it is decided by the OR-gate, not by RevenueCat alone. See also: isPremium,
OR-gate.

### flattenRecord
`src/api/backendClient.js` — the read-side reshaper. Spreads a row's `data` JSONB
to the top level and adds `id`/`created_by`/`user_email`/`created_date`/
`updated_date`, while **dropping** `owner_id`/`owner_email`. So the in-app record
≠ the DB row. With `recordPayload`, it is the de-facto record contract (there are
no TypeScript types for entities).

### generateInitialPlanBundle
`src/lib/generateInitialPlanBundle.js` — the **live** plan-creation path. Writes
profile entities, computes macros (`calcTDEE`), makes **one** overview LLM call,
validates, archives prior active plans, and creates the `AIPlan`. Per-day
workouts/meals are generated lazily later. Not to be confused with
`generateInitialPlans.js`.

### generateInitialPlans.js (legacy)
The disabled old "generate all 7 days up front" module. Only
`buildAnswerContext` and `calcTDEE` are still live (imported by the bundle);
`generateInitialPlans()` itself is a throwing stub. Do not treat it as an active
flow.

### generation_batch_id
A `batch_<ts>_<rand>` id stamped on a plan-generation run, persisted on the
`AIPlan` and propagated to per-day `WorkoutPlan`/`MealPlan` rows (alongside
`source_plan_id`) to link a day's plan back to the master plan it came from.

### house_listings / house board
The **only** standalone, real-column table (`20260622000000_house_board.sql`),
backing the public no-login `executelabs.ca/house` listing board in the `legal/`
submodule. Columns: title, url, gist, neighbourhood, price, beds, baths, tags,
gym, misc, added_by, votes (`{ "<name>": "love"|"maybe"|"pass" }`), created_at.
RLS is **fully permissive** (`using (true)`) — anyone with the anon key reads and
writes; intentional for an unlisted shared board. **Not** an EntityClient entity;
accessed directly via the Supabase client.

### invoke-llm
The Supabase edge function (`supabase/functions/invoke-llm/index.ts`) that proxies
to OpenAI's `/v1/responses` (model gpt-4.1-mini). Reached from JS via
`backend.integrations.Core.InvokeLLM(payload)`. Does full-jitter backoff on
429/5xx honoring `Retry-After`, requires the caller's JWT (401 otherwise),
supports an `input_file` content branch for BYO PDFs, and surfaces a 429 to the
frontend via `error.context.status`.

### isPremium / OR-gate
The premium-unlock truth: premium if **either** the backend `user_subscriptions`
row says premium/active(or trialing) **or** the live RevenueCat
`customerInfo.entitlements.active` set is non-empty. Computed in
`src/hooks/useSubscription.js` (`isPremiumUser(subscription) || liveEntitled`);
the live signal gives instant unlock before the webhook round-trips, the table is
durable truth. **Caveat:** the live half is count-based (any active entitlement),
safe only while "premium" is the only entitlement.

### loading floor
The skeleton/splash a screen holds until the appCache is hydrated
(`cacheReady` from `useCacheHydrated`). Prevents painting the wrong cold-launch
state (e.g. "Build my plan" CTA for a user who has a plan). Driven by the
`appcache:hydration:start`/`done` events. See KEY_FLOWS Flow 4.

### LogModal
`src/components/track/LogModal.jsx` — the portaled framer-motion log sheet shared
by both Track (route) and Home (overlay) so they render the *same* logger. One of
only three components that `createPortal` to `document.body` (needed to escape
the Home pull-to-refresh transform's stacking context).

### lucide-react shim (and recharts shim)
`lucide-react` and `recharts` are **vite-aliased** to hand-maintained re-export
files (`src/lib/lucide-react.js`, `src/lib/recharts.js`). Code imports normally
but resolves to the shim. **Hard gotcha:** importing an icon/member not in the
shim passes lint+tsc but **fails the vite/rollup build**
(`"<X>" is not exported by "src/lib/lucide-react.js"`); fix by appending the
`export { default as <Icon> } from '.../icons/<kebab>.js'` line (alphabetical).

### mergeUpdate
`src/api/backendClient.js` — the update-side helper that merges new fields into
the existing `data` JSONB bag (rather than replacing it) on
`EntityClient.update`.

### optimistic-write + Realtime-reconcile
The single shared write discipline for any UI that both reads a `.subscribe()`
stream and writes the same entity (today only Track + Home, via `saveVitalLog`).
Update in-memory state read-free *before* the await, then let the Realtime echo
reconcile to authority. Needed because Supabase Realtime **echoes a client's own
writes back to itself** after a DB round-trip (hundreds of ms, never instant) —
without optimism you get a flash of stale. See KEY_FLOWS Flow 5, CONVENTIONS
§2.4.

### owner keys / RLS OR-chain
The four interchangeable ownership columns on every entity row — `owner_id`
(uuid) and `owner_email`/`created_by`/`user_email` (email from JWT). A row is
"yours" if **any** matches the caller; `service_role` bypasses entirely. `recordPayload`
duplicates the email into multiple columns precisely to make this OR-chain
reliable. See DATA_MODEL §5.

### paymentClient
`src/lib/paymentClient.js` — the platform-agnostic payment facade. `purchase()`
forks on `getPlatform()`: iOS → RevenueCat/StoreKit, web → Stripe checkout.
Components call this facade, never RevenueCat/Stripe directly.

### PKCE
"Proof Key for Code Exchange" — the OAuth flow the Supabase client is configured
with (`flowType: 'pkce'`, `backendClient.js`). Providers return `?code=`; on iOS
the deep-link handler calls `supabase.auth.exchangeCodeForSession(code)` with the
**bare code value only** (passing the full URL fails the exchange). See KEY_FLOWS
Flow 1.

### platform gate / getPlatform()
`src/lib/platform.js` — `getPlatform()` returns `'ios' | 'android' | 'web'` (plus
`isNative`/`isIOS`/`isAndroid`/`isWeb`). The single switch for native-vs-web
behavior (payments, OAuth redirect style, live StoreKit prices, OTP-vs-magic-link).

### plan_payload
A field inside an `AIPlan`'s `data` JSONB that duplicates the full overview and
holds all BYO state (`byo_targets`, `byo_workout_text`/`byo_meal_text`,
`byo_structured`, `byo_cadence`, `byo_fallback_sides`). App-convention only —
nothing about it is verifiable from the schema.

### PremiumGate / PremiumPaywall
`src/components/premium/` — `PremiumGate.jsx` renders children only if `isPremium`
(else the paywall); `PremiumPaywall.jsx` is the IAP purchase UI (live StoreKit
prices on iOS, Restore Purchases, auto-renew disclosure).

### `recordPayload`
`src/api/backendClient.js` — the write-side inverse of `flattenRecord`: packs an
app object into the DB row shape, duplicating identity (email) into **both** the
wrapper columns and inside `data`. Underpins the RLS OR-chain.

### RevenueCat / StoreKit
The iOS in-app-purchase stack. StoreKit is Apple's payment framework; RevenueCat
wraps it and reports entitlements. The SDK is configured **from JS** via
`ensureConfigured()` in `src/lib/revenuecat.js` (`Purchases.configure`), **not**
natively in AppDelegate — a native configure would leave the Capacitor plugin's
`PurchasesHybridCommon` instance unconfigured and `fatalError` on first
`logIn`/`getOfferings`.

### revenuecatWebhook
The edge function (`supabase/functions/revenuecatWebhook/`) that is the **sole
writer** of `user_subscriptions` rows on the iOS rail — the device never writes
that table. Maps RC event types to `{ plan, status, cancel_at_period_end }`,
authenticates with a bearer secret (constant-time compared), and writes under
`service_role`. The web rail's equivalent writer is `stripeWebhook`.

### saveVitalLog
`src/lib/vitalsLog.js` — owns the whole vital-logging write: read → compute
additive updates → fire `onOptimistic` (post-read, pre-write) → `DailyLog`
update/create → invalidate caches. Returns `{ ok, ... }` (result-object style,
not throw). Shared by Track and Home.

### slash commands (user-invocable skills)
Project chat shorthands that expand to full prompts, run via the Skill tool.
Project-specific ones for Execute:
- **`/deep-review`** — deep analysis of this project's Claude Code sessions;
  fans out specialist analysts and writes a prioritized report.
- **`/ios-sync`** — build the web app and copy it into the iOS project (one clean
  attempt, then diagnose); never starts a dev server. The codified `npm run
  ios:sync` turn.
- **`/log-decision`** — draft a Decision Log entry from the working diff (or a
  topic) and append to `docs/DECISIONS.md` (the "why" log).
- **`/iap-status`** — snapshot the IAP/RevenueCat wiring from the codebase (SDK
  init, package IDs, entitlement, offering, gate, webhook).
- **`/appstore-audit`** — App Store compliance audit + remediation plan.
- **`/xcode-log`** — parse a pasted Xcode console dump, surface real
  errors/faults and RevenueCat/StoreKit lines, end with one next action.
- **`/growth-marketer`** — Instagram ad creative + outreach copy for Execute.
- **`/diverge`** — divergent ideation (flood/invert/transplant) returning a
  scored shortlist.
- Generic Claude Code skills also available: `/simplify`, `/loop`, `/schedule`,
  `/update-config`, `/keybindings-help`, plus Cloudflare/Anthropic-SDK skills.

### `source_plan_id`
A field on a per-day `WorkoutPlan`/`MealPlan` (inside `data`) linking it back to
the master `AIPlan` it was generated from. Convention-only (no DB foreign key).

### structurePastedPlan
`src/lib/plans/structurePastedPlan.js` — the BYO **structuring pre-call**: one
dedicated `InvokeLLM` call that returns either `{ needs_clarification, questions }`
or `{ structured: { workout?, nutrition? } }`, and derives an activity level +
cadence that feed `calcTDEE` *before* the overview call.

### SWR (stale-while-revalidate)
The cache read pattern: paint instantly from cache, check freshness, kick a
background network refresh that writes back. Used both by appCache (durable) and
by the lightweight module-level TTL caches (e.g. `_cachedSubscription` with
`CACHE_TTL_MS`). Not a library — a hand-rolled idiom. See CONVENTIONS §2.3.

### `TABLES` map
The single Entity-name → DB-table-name source in `src/api/backendClient.js` (19
entities). Add new entities here; never hardcode table names elsewhere.

### TanStack Query
`@tanstack/react-query` — used for server-state in parts of the app
(`queryClientInstance` in `src/lib/query-client.js`), but **not** universal; many
lib modules use module-level TTL caches instead. Match the file you're editing.

### user_subscriptions / UserSubscription
The entity/table holding subscription state (`plan`, `status`,
`cancel_at_period_end`). Its `user_id` column holds an **email**, not a uuid.
Written **only** server-side (`revenuecatWebhook` on iOS, `stripeWebhook` on web);
the device only reads it (`subscription.js`). Source of one half of the OR-gate.

### vitals / DailyLog
A "vital" is a tracked daily metric (sleep, water, steps, mood, energy, weight,
calories burned, habits — `ALL_CATEGORIES` in `src/components/track/categories.js`).
They are logged into the per-day `DailyLog` entity. `ADDITIVE_FIELDS`
(steps/sleep/water) accumulate; others overwrite.

### withBackoff
`src/lib/withBackoff.js` — the repo's DB-side retry helper: a full-jitter
exponential-backoff loop (retries 5, base 300ms, cap 8s, deadline 30s) that
honors `Retry-After` and retries on 429/503/5xx (matched by status **and** error
text, since PostgREST 429s don't always surface a clean numeric status). Wraps
every per-day DB op in the multi-day workout **and** nutrition build paths; the
repo-wide standard for multi-day plan builds.

### weekly_overview
The 7-day array inside an `AIPlan`'s overview (`data`), one entry per day with a
`day_type`/`workout_needed` and (for BYO) `byo_session`/`byo_meal_focus`.
Per-day generation reads the matching day from here. App-convention shape only.
